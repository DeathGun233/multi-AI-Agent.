# 第 3 步：前端工作台升级

## 这一步做了什么

这次升级把原来只有“模板选择 + JSON 输入 + 运行结果”的单页 Demo，升级成了真正的工作流控制台。

当前前端已经具备这三块核心能力：

- 运行历史页
- 人工审核区
- 执行轨迹区

## 新增的页面能力

### 1. 运行历史

现在页面会展示最近运行过的工作流列表，包括：

- 工作流类型
- 当前状态
- 当前步骤
- 更新时间
- 运行 ID

点击任意一条历史记录，就能在右侧查看完整详情。

### 2. 人工审核

新增了独立审核队列：

- 页面会列出所有 `waiting_human` 状态的任务
- 可以直接选中待审核任务
- 可以填写备注后执行“人工通过”或“人工驳回”

这部分配套新增后端接口：

- `GET /api/workflows/review-queue`
- `POST /api/workflows/{id}/review`

### 3. 执行轨迹

右侧详情区会展示：

- LangGraph 节点流转日志
- 每个 Agent 的执行信息
- 工具调用记录
- 最终结构化结果 JSON

同时还展示图结构：

- 节点列表
- 边关系
- 运行时类型 `langgraph`

## 当前页面布局

控制台分为三列：

### 左侧

- 工作流模板中心
- 审核队列

### 中间

- 新建工作流表单
- 最近运行历史

### 右侧

- 当前任务详情
- 审核动作
- LangGraph 图结构
- 执行轨迹
- 完整结果 JSON

## 为什么这一步重要

前两个阶段虽然已经有：

- 真实模型
- MySQL / Redis 兼容
- LangGraph 编排

但如果没有一个能“看见历史、轨迹和审核动作”的界面，展示效果还是偏开发视角，不够像成品系统。

这一步的价值在于把系统从“可跑的后端”升级成“可展示、可操作、可讲述的产品形态”。

## 新增接口

除了之前已有接口，这一步新增：

- `GET /api/workflows/review-queue`
- `POST /api/workflows/{run_id}/review`

## 验证方式

这一步新增验证点：

- 首页能渲染控制台结构
- 审核队列接口能返回待处理任务
- 对等待人工接管的任务可以执行人工通过
- 人工审核日志会写入运行记录

## 当前边界

这一步还没有继续做：

- 独立路由的多页面系统
- 图形化时间轴动画
- 图节点高亮联动
- 审核权限控制

这些可以在后面继续增强，但当前版本已经足够作为一个完整项目演示页。
