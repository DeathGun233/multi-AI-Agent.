# 第 4 步：多页面、执行时间线与权限升级

## 这一步做了什么

这次升级把原本偏 Demo 形态的单页控制台，进一步做成了更接近企业后台的工作台版本，重点落了三件事：

- 拆成独立多页面
- 加入图形化执行时间线
- 增加登录态与审核权限

## 多页面结构

当前前端页面已经拆成以下几个独立入口：

- `/login`
  用于账号登录和建立会话
- `/dashboard`
  用于发起工作流、查看整体指标、快速跳转最近运行和审核积压
- `/runs`
  用于查看全部运行历史
- `/reviews`
  用于处理 `waiting_human` 队列
- `/runs/{id}`
  用于查看单个任务的完整详情

这样一来，系统展示方式就不再是“一个页面塞下所有东西”，而是更接近真实后台产品。

## 图形化执行时间线

任务详情页新增了可视化时间线区域。每条运行日志会映射成一个节点，展示：

- 执行代理
- 执行动作
- 时间点
- 阶段耗时
- 是否有工具调用

这个时间线不是简单的日志堆叠，而是把执行路径变成可读的图形结构，更适合演示：

- Planner 做了什么
- Operator 在什么时候调了工具
- Analyst、Content、Reviewer 如何串起来
- 最终是自动完成还是转人工接管

## 登录态设计

当前版本采用轻量会话方案：

- 使用签名 Cookie 保存登录态
- 登录成功后进入对应工作台
- 后端会校验 Cookie 是否被篡改

这套实现适合当前项目阶段：

- 不依赖外部认证服务
- 本地开发和演示成本低
- 后续可平滑替换成企业 SSO、JWT 或正式用户系统

## 角色与权限

系统内置 4 类演示角色：

- `viewer`
  只能查看页面和接口
- `operator`
  可以发起工作流，但不能审核
- `reviewer`
  可以查看并处理审核队列
- `admin`
  拥有发起和审核两类权限

当前权限控制重点放在两类动作上：

- 发起工作流
- 人工审核通过 / 驳回

这样能更真实地体现企业流程里的职责边界。

## 新增接口与页面动作

这一步补充了：

- `GET /login`
- `POST /login`
- `POST /logout`
- `GET /dashboard`
- `GET /runs`
- `GET /reviews`
- `GET /runs/{id}`
- `GET /api/session`

并新增了页面表单动作：

- `POST /dashboard/run`
- `POST /runs/{id}/review/form`

## 为什么这一步重要

前几步已经解决了：

- 真实模型接入
- MySQL / Redis 兼容持久化
- LangGraph 状态流编排

但如果没有登录、权限和多页面结构，项目更像“技术演示台”，而不是“可交付产品”。

这一步的价值在于让项目具备更强的成品感：

- 能展示角色边界
- 能展示操作路径
- 能展示审核运营流程
- 能展示单次任务的可视化执行轨迹

## 当前边界

这一步还没有继续做：

- 正式数据库用户表
- 密码哈希与重置流程
- OAuth / SSO
- 细粒度菜单权限
- 更复杂的时间线交互动画

但对于当前项目目标来说，已经足够支撑面试展示、项目答辩和后续继续扩展。
