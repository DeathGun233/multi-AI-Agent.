# 第 5 步：数据库认证与密码哈希升级

## 这一步做了什么

这一轮把原来偏演示性质的内存账号体系，升级成了更接近正式项目的认证基础版：

- 新增数据库用户表
- 启动时自动初始化默认账号
- 密码不再明文保存，而是做哈希存储
- 登录校验从内存字典切换为数据库查询

## 新增的数据表

本次新增 `user_accounts` 表，主要字段包括：

- `username`
- `password_hash`
- `display_name`
- `role`
- `is_active`
- `created_at`
- `updated_at`

这样之后如果要继续扩展，就可以在同一套结构上增加：

- 用户管理页面
- 禁用 / 启用账号
- 修改密码
- 审核人分组
- 更细粒度的权限模型

## 密码哈希方案

当前版本没有继续使用明文密码，而是切换到：

- `PBKDF2-SHA256`

保存格式类似：

- `pbkdf2_sha256$轮数$salt$digest`

这样做的意义是：

- 避免数据库里直接出现明文密码
- 提高密码泄露后的攻击成本
- 后续可以继续平滑升级到更强的密码策略

## 启动初始化逻辑

应用启动时会检查数据库里的用户表。

如果默认账号不存在，就自动插入以下种子用户：

- `admin`
- `reviewer`
- `operator`
- `viewer`

这些账号仍然保留，是为了让项目本地演示和测试保持低成本，但它们现在已经是数据库账号，而不是只存在于代码内存里。

## `FLOWPILOT_USERS_JSON` 的角色变化

之前这个变量更接近“直接定义运行时账号”。

现在它的意义调整为：

- 作为数据库种子用户输入

也就是说，它只负责初始化数据库里的账号，不再作为运行时认证的直接来源。

## 为什么这一步重要

第四步虽然已经有：

- 登录页
- 会话 Cookie
- 角色权限

但如果账号本身不落库，还是偏 Demo。

这一步补上之后，项目在认证层面更像一个真实系统：

- 用户是数据库实体
- 密码是哈希而不是明文
- 会话读取后会回查数据库用户状态
- 后续接 JWT、SSO、用户管理页都有明确延展点

## 当前边界

这一版还没有继续做：

- 密码重置流程
- 注册流程
- JWT Access Token / Refresh Token
- OAuth / 企业 SSO
- 审计日志
- 权限点级别的 RBAC 模型

但作为当前项目的下一步，这是非常关键的一层基础建设。
